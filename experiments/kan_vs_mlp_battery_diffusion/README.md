# KAN vs MLP - battery diffusion

Can a Kolmogorov-Arnold Network solve a differential equation as a
physics-informed neural network (PINN) at least as well as a standard MLP
with the same parameter budget - and what does it cost to find out?

This experiment trains both architectures to solve the same 1D lithium-ion
diffusion equation (`generators/diffusion_1d/`, framed as a battery
electrode charging) purely from the physics - the PDE residual plus
boundary/initial conditions, no labeled data - and checks each trained
model against a finite-difference reference solution it never saw during
training.

## Setup

- **Generator**: `generators/diffusion_1d/` - Crank-Nicolson reference
  solution. D=4e-14 m²/s, L=6e-6 m, c0=15000 mol/m³, applied surface flux
  J=2.5e-5 mol/(m²s), t_end=1200 s (20 min charge). Fourier number = 1.333.
- **MLP**: `[2, 20, 20, 20, 1]`, tanh - **921 parameters**
- **KAN**: `[2, 6, 6, 6, 1]`, grid_size=6, cubic splines - **900 parameters**
  (own implementation, see `models/kan/README.md`)
- Both trained on the *identical* physics-informed loss (PDE + BC + IC
  residual, dimensionless), the *identical* collocation points per seed,
  and the *identical* optimizer/epoch budget - see `config.yaml` for every
  number.
- **5 seeds** (0-4), each reseeding both the shared collocation points and
  each architecture's own initialization, so results reflect sensitivity to
  initialization rather than a single lucky/unlucky run.

## Finding

| Architecture | Params | Final loss | Rel. L2 error vs FD | Mean abs error [mol/m³] | Wall time [s] | Time/epoch [ms] |
|---|---|---|---|---|---|---|
| KAN | 900 | 5.95e-4 ± 1.19e-4 | **4.66e-3 ± 9.21e-4** | 11.0 ± 4.2 | 503.9 ± 37.1 | 167.97 |
| MLP | 921 | 1.77e-3 ± 3.07e-4 | 7.26e-3 ± 1.09e-3 | 16.2 ± 3.5 | **30.8 ± 9.1** | 10.28 |

(from `outputs/comparison_table.md`, generated by `run.py` from
`outputs/phase3_results.json` - see that file for the full per-seed,
per-epoch history behind these numbers)

**KAN wins on accuracy, consistently.** With essentially the same parameter
count, KAN reaches ~1.6x lower error than the MLP - and not by luck: KAN's
*worst* seed (rel. L2 = 5.6e-3) still beats MLP's *best* seed (5.7e-3), no
overlap across all 5 runs (`outputs/error_distribution.png`).

**KAN loses on training cost, badly.** ~16x slower wall-clock time for the
same epoch budget, despite having *fewer* parameters than the MLP. The cost
isn't model size - it's evaluating B-spline basis functions on every edge,
every forward pass, which is far more expensive per FLOP than a plain
matrix multiply (`outputs/cost_accuracy_tradeoff.png` shows the two
architectures as two cleanly separated clusters, no ambiguity about the
trade-off).

**Both fit the real physics well.** `outputs/solution_fit_comparison.png`
overlays each architecture's best-of-seeds prediction against the FD
reference at four points in time - both track it closely; the visible
discrepancy is concentrated at `t=0`, where the initial condition is a
*soft* constraint in the loss (not hard-enforced), and the MLP strays from
it slightly more than the KAN does.

Caveat worth keeping in mind before generalizing: this KAN implementation
is not optimized for speed (the Cox-de Boor recursion isn't fused/
vectorized as aggressively as it could be). The ~16x factor would likely
shrink with a more optimized implementation or a GPU - but the qualitative
result (KAN more accurate, MLP much cheaper per epoch) is a real,
consistent property of evaluating splines vs. matrix multiplies, not an
artifact of this specific codebase.

## Reproduce

```bash
pip install -r ../../requirements.txt
python run.py
```

`run.py` is the single entry point: it (1) solves the reference equation
with the physical parameters in `config.yaml`, (2) trains both
architectures across every configured seed, (3) evaluates each trained
model against the reference, (4) writes `outputs/comparison_table.csv/.md`,
`outputs/results_raw.csv`, `outputs/phase3_results.json`, the four PNG
figures in `outputs/`, and the trained weights to `checkpoints/`.

Everything reproducible lives in `config.yaml` - physical constants,
architecture sizes, seeds, epochs, learning rate. Change a value and rerun
to reproduce a variant.

**Expected runtime**: ~45-60 minutes on CPU for the default config (5 seeds
x 2 architectures x 3000 epochs). CPU is used deliberately for training
determinism (see `config.yaml`); these networks are small enough that this
isn't a real bottleneck.
